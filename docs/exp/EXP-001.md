# EXP-001: The 12 GB Agent v0.1 — First Characterization of Local + Ollama-Cloud Models on PBS v0.1

Date created: 2026-05-25
Status: planned
Pre-registered: (commit SHA filled by `lab exp register` at registration time)

## Hypothesis

Four pre-registered hypotheses, all evaluated at greedy decoding (T=0, top_p=1.0, max_tokens=1024), N=8 seeds per cell, on PBS v0.1 (24 tasks: 8 math-reasoning + 8 format-following + 8 knowledge-recall).

- **H1 — Reasoning gap (cloud beats local on math):** On PBS math-reasoning, `gpt-oss-120b-cloud` achieves mean pass@1 at least **15 percentage points higher** than the best of the four-locals `{qwen3-14b-q4, llama3.1-8b-q4, gemma3-12b-q4, phi4}`.

- **H2 — Knowledge near-parity:** On PBS knowledge-recall, the gap between `gpt-oss-120b-cloud` and the best local model is **≤ 10 pp** on mean pass@1. (Knowledge tasks should be less sensitive to compute since they're recall-bound, not reasoning-bound.)

- **H3 — Reasoning-mode advantage on format-following:** On PBS format-following, `qwen3-14b-q4` (reasoning enabled by default) achieves mean pass@1 at least **20 pp higher** than each of `gemma3-12b-q4` and `llama3.1-8b-q4` (no reasoning mode).

- **H4 — Reliability cliff:** Reliability ratio (mean `pass^8 / pass@1` averaged across cells) will be **≥ 0.75 for `gpt-oss-120b-cloud`** AND **≤ 0.50 for at least one local-12GB model**. Frontier cloud models should be consistently right when they're right; small local models should "fluke into" wins more often.

These hypotheses are **mutually independent**: each can be confirmed or refuted on its own evidence. We pre-commit to reporting all four verdicts in F-003 regardless of which way they fall.

## Why this matters

This is the lab's first real research output. The questions it answers are practical:

1. *Which local 12-GB model do I reach for first?* (informs default model selection in all future lab work)
2. *When is it worth burning cloud budget on a 120B-class model?* (informs the 95/5 local/cloud split heuristic from RESEARCH_OPS_PLAN)
3. *Does qwen3's reasoning mode actually help on small structured tasks, or is the extra latency wasted?* (cross-checks the F-001/F-002 observations with a wider task surface)
4. *Where's the reliability floor for 12-GB models?* (justifies — or refutes — the N≥8 discipline for production-bound decisions)

## Method

### Models (6)

| litellm_id | Backend | VRAM | Reasoning mode |
|---|---|---|---|
| `qwen3-14b-q4` | local Ollama | 9.3 GB | yes (default on) |
| `llama3.1-8b-q4` | local Ollama | 4.9 GB | no |
| `gemma3-12b-q4` | local Ollama | 8.1 GB | no |
| `phi4` | local Ollama | 9.1 GB | no (but trained on reasoning data) |
| `gpt-oss-20b-cloud` | Ollama Cloud Pro | n/a (cloud) | no |
| `gpt-oss-120b-cloud` | Ollama Cloud Pro | n/a (cloud) | no |

Notes:
- The quantization axis is **dropped from v0.1**: `qwen3:14b` ships only Q4_K_M on the public Ollama registry (verified 2026-05-25). The quant question moves to **EXP-002**, using a model family that publishes multiple quants (likely `llama3.1:8b` with Q4_K_M / Q5_K_M / Q8_0 / fp16 all available).
- `phi4` is included to test whether reasoning-trained models without explicit reasoning-mode tokens approach qwen3's behaviour.

### Config (1)

```yaml
name: greedy-1024
temperature: 0.0
top_p: 1.0
max_tokens: 1024
scaffold: single_turn
```

### Tasks

All 24 tasks in PBS v0.1. No new tasks for this experiment — we have already pre-registered the suite via Phase 3.

### Seeds (8)

`[1, 2, 3, 4, 5, 6, 7, 8]` per protocol [`reliability-sweep.md`](../protocols/reliability-sweep.md).

### Total cells

24 tasks × 6 models × 1 config × 8 seeds = **1,152 runs**.

### Evaluators (pre-registered)

Deterministic (free, local; applied to every done run):
- `exact_match` — for tasks with `gold_answer` and `rubric.type=exact_match`
- `regex_match` — for tasks with `rubric.type=regex`
- `not_empty` — sanity
- `latency_under` (10 s default) — operational signal
- `tokens_under` (500 default) — operational signal
- `cost_under` — applies only when cost_usd is populated (cloud); 0.05 USD threshold

LLM-judge (paid, cloud-budgeted):
- `llm_judge_quality` — judge=`gpt-oss-20b-cloud`, applied to all 1152 runs
- Oracle slice: 10% sampled uniformly at random with `seed=42`, re-judged by `gpt-oss-120b-cloud` per [`protocols/judge-calibration.md`](../protocols/judge-calibration.md)
- Cohen's kappa + Pearson r between cheap-judge and oracle reported in F-003

### Statistics

Per protocol `reliability-sweep.md`:
- `pass@1`, `pass^4`, `pass^8` for each (model, task) cell
- bootstrap 95% CI on cell pass rates (n_resamples=2000)
- reliability ratio = mean(pass^8) / mean(pass@1) across cells, per model
- For H1/H2/H3 deltas: Welch's t-test on per-task mean scores, report p-value + effect size

## Success / failure criteria

Each hypothesis is judged by the **pre-registered rule**, applied AFTER the sweep + scoring complete. We do not look at per-run results before deciding the verdict.

- **H1 confirmed** ⇔ `mean_pass_at_1(gpt-oss-120b-cloud, math-reasoning) − max(mean_pass_at_1(L, math-reasoning) for L in locals) ≥ 0.15` AND Welch's t-test p < 0.05.
- **H2 confirmed** ⇔ same delta on knowledge-recall is **≤ 0.10** (and best local exists with a passing CI).
- **H3 confirmed** ⇔ `qwen3-14b-q4`'s mean pass@1 on format-following is `≥ 0.20` higher than each of `gemma3-12b-q4` and `llama3.1-8b-q4`, both with p < 0.05.
- **H4 confirmed** ⇔ `reliability_ratio(gpt-oss-120b-cloud) ≥ 0.75` AND `min(reliability_ratio(L) for L in locals) ≤ 0.50`.

Any failure modes (e.g. >5% cells error out, judge agreement with oracle < r=0.6 / kappa<0.4) are escalated in F-003 rather than swept under the rug.

## Confounders to control

- **Outer-loop-by-model** sweep ordering to minimise cold-start variance (Phase 1 playbook).
- **One config**: no temperature/top_p variation in v0.1. This isolates the model and quant axes.
- **Same seed schedule** across all cells.
- **Same prompt**: no system prompt; user-message only. Removes prompt-engineering as a variable for v0.1.
- **No quant axis**: deferred to EXP-002 (qwen3:14b doesn't ship multiple quants on Ollama registry as of 2026-05-25).

## Kill criteria

- If any single infra component (Postgres, MinIO, LiteLLM, Ollama) prevents >5% of cells from completing, stop and fix before continuing the sweep.
- If Ollama Cloud Pro tier returns sustained 429s indicating quota exhaustion, **pause for 5 hours** to let the session window reset and resume.
- If judge agreement with oracle on the 10% slice is below r=0.6 / kappa=0.4, **don't apply the cheap judge to the bulk** — re-do the whole judge slice with the 120b model directly, accepting the higher budget burn.

## Pre-mortem

Imagine it's 3 days from now and EXP-001 was a methodological failure. What plausibly went wrong?

1. **Qwen3 reasoning ate the 1024 token budget on N% of cells** → not_empty pass rate drops; pass@1 corrupted. Mitigation: report not_empty rate per model in F-003; if >10% empty for any cell, document and re-run that cell only with `max_tokens=2048`.
2. **Cloud quota exhausted mid-sweep** → quota tracker should fire alert at 80%. We monitor + can pause. The sweep is resumable so this is recoverable.
3. **Judge disagreement is high** → kill criteria above triggers oracle-on-all instead of cheap-judge-with-calibration.
4. **A task is contaminated for one model but not others** → flagged by abnormally high pass@1 on one model vs others (e.g. 8/8 on a hard task when others get 0/8). Discussed in F-003 caveats; no PBS task changes mid-experiment.
5. **GPU thermal throttling** → latency rows spike. Mitigation: capture nvidia-smi in manifest; F-003 reports correlation between time-of-day and latency to detect this.
