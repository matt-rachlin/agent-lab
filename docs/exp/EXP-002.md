# EXP-002: The 12 GB Agent v0.2 — First Tool-Use Characterization

Date created: 2026-05-26
Status: planned
Pre-registered: (commit SHA filled by `lab exp register` at registration time)

## Question

Do local 12 GB models survive a real multi-turn tool-use loop? EXP-001 / F-003
established their single-turn behaviour on PBS-v0.1. This experiment is the
first measurement of what happens when the *same* class of models is dropped
into an agent harness: MCP tools, a sandboxed environment, multi-turn
planning, and end-state grading rather than string-match. We pre-commit four
hypotheses about how locals will compare to Ollama Cloud Pro frontier models
on tool-call accuracy, multi-turn reliability, and cost-vs-latency scaling.

## Hypothesis

Four pre-registered hypotheses, all evaluated at greedy decoding
(`temperature=0.0`, `top_p=1.0`, `max_tokens=1024`), N=8 seeds per cell, on
PBS-Agent v0.1 (12 hand-curated tool-use tasks: 3 fs + 3 code + 2 shell +
2 http + 2 multi-domain). Each cell runs inside a Podman + gVisor sandbox.

- **H1 — Cloud tool-call accuracy:** mean `tool_correctness` (over all 12
  tasks, all 8 seeds) for the **3 cloud models**
  (`gpt-oss-20b-cloud`, `glm-5.1-cloud`, `gpt-oss-120b-cloud`) is **≥ 0.60**.

- **H2 — Local tool-call accuracy:** mean `tool_correctness` (over all 12
  tasks, all 8 seeds) for the **2 local models**
  (`qwen3-14b-q4`, `llama3.1-8b-q4`) is **≥ 0.40**.

- **H3 — Multi-turn reliability cliff:** for **at least one** local model,
  `pass⁸ / pass¹` on the `end_state` scorer drops **below 0.70** on tasks
  with `max_turns ≥ 3`. (All PBS-Agent v0.1 tasks have `max_turns ≥ 4`, so
  the cliff condition reduces to "on the full suite". The structural
  expectation is that small models *fluke* into wins on a single seed but
  cannot maintain that across eight independent trajectories.)

- **H4 — Cost/turn scaling:** between `gpt-oss-20b-cloud` and
  `gpt-oss-120b-cloud`, **normalised cost per turn grows faster than
  normalised mean turn latency**. Formally: with `c = mean cost_usd per turn`
  and `l = mean wall_time_sec per turn`, the ratios across the two models
  satisfy `(c_120b / c_20b) ≥ 1.5 × (l_120b / l_20b)`. (Cloud GPU minutes
  cost is amortised into the Pro subscription; the comparison uses
  per-call cost weights from `lab.quota._MODEL_WEIGHTS` as a proxy when
  metered cost is $0.)

These hypotheses are mutually independent: each can be confirmed or refuted
on its own evidence. We pre-commit to reporting all four verdicts in F-005
regardless of which way they fall.

## Why this matters

This is the lab's first tool-use measurement. EXP-001 told us nothing about
multi-turn behaviour; F-003 explicitly punted the question. The answers
inform:

1. *Is local-first feasible for agentic loops?* (decides whether to spend
   Phase 7+ on local-model agent tooling or just route everything to cloud)
2. *Which cloud model is the right default when local can't handle it?*
   (3-way cloud bake-off: 20B vs glm-5.1 vs 120B)
3. *Does the budget-cliff predicted by H4 actually show up at our scale?*
   (informs the 95/5 local/cloud heuristic at the agentic granularity, not
   just single-turn)
4. *Where's the multi-turn reliability floor?* (justifies — or refutes —
   N≥8 for agentic decisions; agentic variance is structurally higher than
   single-turn variance because tool errors compound)

## Method

### Models (5, in this order)

| litellm_id | Backend | VRAM / tier | Notes |
|---|---|---|---|
| `qwen3-14b-q4` | local Ollama | 9.3 GB | reasoning **disabled** via API-level `think: false` per user direction 2026-05-26 (see "Confounders"); reasoning-ON ablation deferred to EXP-002b |
| `llama3.1-8b-q4` | local Ollama | 4.9 GB | no reasoning mode |
| `gpt-oss-20b-cloud` | Ollama Cloud Pro | cloud | cheapest cloud reference |
| `glm-5.1-cloud` | Ollama Cloud Pro | cloud | newly registered (commit `9053216`); first lab use |
| `gpt-oss-120b-cloud` | Ollama Cloud Pro | cloud | upper bound for "spend more, do better" |

### Config (1)

```yaml
name: greedy-1024
temperature: 0.0
top_p: 1.0
max_tokens: 1024
scaffold: single_turn   # runner dispatches to agent path on max_turns>1
```

`scaffold: single_turn` is the metadata value carried on every config.
Actual dispatch to the agent loop happens automatically in
`lab.sweep.runner.execute_cell` when the task's `max_turns > 1` (all 12
PBS-Agent v0.1 tasks satisfy this).

### Tasks (12)

The full PBS-Agent v0.1 suite (`tasks/pbs-agent-v0.1/`, suite name
`pbs-agent-v0.1`):

| Category | Count | Slugs |
|---|---|---|
| fs | 3 | `fs-read-and-copy`, `fs-grep-extract-and-write`, `fs-write-csv-summary` |
| code | 3 | `code-read-and-explain`, `code-write-and-execute`, `code-find-and-fix-bug` |
| shell | 2 | `shell-count-lines`, `shell-pipeline-extract` |
| http | 2 | `http-fetch-and-extract`, `http-fetch-and-count` |
| multi | 2 | `multi-words-and-hash`, `multi-db-self-check` |

All tasks pre-registered in Phase 6f (commit `eee48d6`). Per-task
`max_turns` ranges 4-6; per-task `tool_budget` ranges 4-8.

### Seeds (8)

`[1, 2, 3, 4, 5, 6, 7, 8]` per protocol [`reliability-sweep.md`](../protocols/reliability-sweep.md).

### Total cells

12 tasks × 5 models × 1 config × 8 seeds = **480 runs**.

### Evaluators (pre-registered)

Deterministic, applied to every done run:

- `end_state` — task `success_predicate` over post-run `/workspace` snapshot
  and/or DB state (every task carries one).
- `tool_correctness` — model called `target_tool` declared in
  `task.rubric.tool_call` (every task carries one).
- `budget_respected` — `actual_turns ≤ max_turns AND tool_call_count ≤
  tool_budget` (operational signal).

LLM-judge (cloud-budgeted, **selective**):

- `trajectory_judge` — applied **only** to tasks with
  `success_predicate.include_judge: true`. In the v0.1 suite this is
  the single task `code-read-and-explain` (judge model
  `gpt-oss-120b-cloud`, as set on the task). One judged task × 5 models ×
  8 seeds = **40 judge calls**, not 480.

This selective-judge discipline is the explicit reason `include_judge` was
added to the task schema in Phase 6f — blanket-judging 480 trajectories is
not worth the budget or the noise.

### Statistics

Per protocol `reliability-sweep.md`:

- `pass@1`, `pass^4`, `pass^8` per (model, task) cell on `end_state` and
  `tool_correctness`.
- Bootstrap 95 % CI on cell pass rates (n_resamples=2000).
- Per-model mean `tool_correctness` for H1/H2 verdicts.
- Per-model reliability ratio `mean(pass^8) / mean(pass@1)` on `end_state`,
  restricted to tasks with `max_turns ≥ 3` (= all 12), for H3.
- Per-model mean per-turn cost weight (from `lab.quota._MODEL_WEIGHTS`) and
  mean per-turn wall time, for H4.

## Success / failure criteria

Each hypothesis is judged by the pre-registered rule, applied AFTER the
sweep + scoring complete. No peeking.

- **H1 confirmed** ⇔
  `mean(tool_correctness over {gpt-oss-20b-cloud, glm-5.1-cloud,
  gpt-oss-120b-cloud} × 12 tasks × 8 seeds) ≥ 0.60`.
- **H2 confirmed** ⇔
  `mean(tool_correctness over {qwen3-14b-q4, llama3.1-8b-q4} × 12 tasks ×
  8 seeds) ≥ 0.40`.
- **H3 confirmed** ⇔
  `∃ L ∈ {qwen3-14b-q4, llama3.1-8b-q4} such that
  mean_over_tasks(pass^8(L) / pass^1(L)) < 0.70` on the `end_state` scorer.
  (Cells with `pass^1 = 0` are excluded from the ratio to avoid div-by-zero;
  if a model has fewer than 6/12 tasks with nonzero `pass^1`, H3 is
  reported as "undefined" rather than confirmed/refuted.)
- **H4 confirmed** ⇔
  `(cost_per_turn(120b) / cost_per_turn(20b)) ≥ 1.5 ×
  (latency_per_turn(120b) / latency_per_turn(20b))`, both ratios computed
  from per-cell means then averaged uniformly across the 12 tasks.

Any failure modes (>5 % cells errored, sandbox failure rate >10 %, judge
agreement implausible) are escalated in F-005 rather than swept under
the rug.

## Confounders to control

- **Outer-loop-by-model** sweep ordering to minimise cold-start variance
  (Phase 1 playbook).
- **One config**: no temperature / top_p / max_tokens variation. This
  isolates the model and tool-loop axes.
- **Same seed schedule** across all cells.
- **qwen3 runs with `think: false`**: EXP-001b showed `/no_think` in the
  system prompt does NOT disable reasoning — only the API-level
  `think: false` Ollama knob does (commit `ae3f1db`). Per user direction
  on 2026-05-26 we disable reasoning for `qwen3-14b-q4` in EXP-002 via
  the per-model override `model_defaults.qwen3-14b-q4.extra.think:
  false`. This holds the EXP-001b finding (reasoning is net-negative for
  qwen3 on PBS-v0.1) constant when moving to PBS-Agent v0.1. The
  reasoning-ON comparison is deferred to **EXP-002b** (parallel to
  EXP-001b's structure relative to EXP-001).
- **One sandbox image**: `sandbox_image_hash` is captured per run; we
  refuse to start the sweep if the image hash drifts mid-sweep.
- **No prompt variation**: tasks ship with `task.system` and `task.input`;
  no model-defaults overrides applied.
- **Per-cell sandbox isolation**: every cell spins up a fresh container
  (Podman `--runtime=runsc`, `--network=none` default), so cross-cell
  state pollution is impossible by construction.

## Kill criteria

- **Cell error rate ≥ 5 %**: stop, triage, do NOT continue silently. Most
  likely causes ranked: (1) sandbox boot timeout, (2) Inspect harness
  exception, (3) MCP stdio handshake failure, (4) LiteLLM proxy 5xx storm,
  (5) Ollama Cloud quota exhaustion.
- **Sandbox failure rate ≥ 10 %** (specifically: container failed to start
  or `runsc` errored): abort and escalate to gVisor diagnostics.
- **Ollama Cloud quota exhaustion** (sustained 429s): pause for 5 hours
  per Pro session reset window, then resume from `--resume`.
- **Judge disagreement implausible** on the 40 trajectory-judge calls:
  document in F-005; do NOT iterate judge prompts post-hoc.

## Pre-mortem

It's 3 days from now and EXP-002 was a methodological failure. What
plausibly went wrong?

1. **gVisor + Fedora 43 + SELinux interaction** breaks N% of cells with
   `runsc` errors. Mitigation: `runsc` was smoke-tested in 6b commit
   `5c364cc`; we recapture sandbox_image_hash per run and abort on
   sustained sandbox failure.
2. **Qwen3 reasoning blows the 1024 budget on tool-use cells** (same
   failure mode as EXP-001 H3). **Mitigated**: reasoning disabled via
   `think: false` per user direction 2026-05-26. EXP-002b is the
   reasoning-ON ablation (parallel to EXP-001b relative to EXP-001).
3. **LiteLLM tool-call passthrough broken on glm-5.1-cloud** (first lab
   use of this route). Mitigation: tool-call passthrough verified in
   6b/6c smoke before this experiment; failures here surface as
   `tool_correctness = 0` for glm-5.1 specifically and are diagnosable.
4. **Cloud quota wall mid-sweep** with 480 cells / 5 models / Pro tier
   (~50-150 cloud calls/day comfortable). Mitigation: 3 cloud models × 12
   tasks × 8 seeds = 288 cloud cells; we plan for sweep to span 2
   calendar days and resume cleanly.
5. **A task is contaminated** (one model nails a hard task while others
   fail). Documented in F-005 caveats; no PBS-Agent task changes
   mid-experiment.

## Budget estimate

Worst-case sizing of the sweep before launch.

### Token volume (worst case)

Assumption: average 4 turns × 800 tokens per turn (input + output combined)
per cell. This rounds up most cells — `max_turns` ranges 4-6, real-world
turn averages run lower.

```
per cell:     4 turns × 800 tok = 3,200 tokens
per (model, task, seed) cell:                3,200 tokens
total cells:  5 models × 12 tasks × 8 seeds =       480
total tokens: 480 × 3,200 =                  1,536,000 tokens ≈ 1.6 M
```

Per-model split (each model gets 12 × 8 = 96 cells × 3,200 tok = 307,200
tok ≈ **0.31 M tokens / model**, worst case).

Trajectory judge (only `code-read-and-explain`, judge model
`gpt-oss-120b-cloud`): 1 task × 5 models × 8 seeds = 40 judge calls. Each
judge call ingests a trajectory (~5,000 tokens worst case) and emits ≤200
tokens. Worst case judge tokens: 40 × 5,200 = **0.21 M tokens** on
`gpt-oss-120b-cloud`. (Folded into the gpt-oss-120b-cloud row of the cost
table below.)

### Cash cost

Ollama Cloud Pro is a flat-rate subscription (no metered overage). Per the
RESEARCH_OPS_PLAN heuristics and the same convention used in EXP-001:

| Model | Backend | Token volume (worst case) | Cash $ |
|---|---|---|---|
| `qwen3-14b-q4` | local | 0.31 M | $0 |
| `llama3.1-8b-q4` | local | 0.31 M | $0 |
| `gpt-oss-20b-cloud` | Ollama Cloud Pro | 0.31 M | $0 (Pro subscription) |
| `glm-5.1-cloud` | Ollama Cloud Pro | 0.31 M | $0 (Pro subscription) |
| `gpt-oss-120b-cloud` | Ollama Cloud Pro | 0.31 M generation + 0.21 M judge = 0.52 M | $0 (Pro subscription) |
| **Total** | — | **~1.75 M tokens** | **$0** |

Local electricity (RTX 3080 Ti at full draw for the local share): ≈ **$0.20**.

### Cloud-call budget (quota proxy)

Pro tier comfortable headroom on 20B/120B-class models is ~50-150 calls/day.
This sweep will issue:

```
cloud generation calls: 3 cloud models × 12 tasks × 8 seeds = 288 cells
                        × 4 turns avg                      = 1,152 cloud invocations
                        (288 cells × 4 turn avg)

trajectory judge:                                                    40 calls
                                                                  -------
total cloud invocations:                                          1,192
```

That's ~3-5 days of comfortable Pro budget if run sequentially in one
calendar day; **plan for the sweep to span 2 calendar days** with one
overnight pause if quota tightens. `--resume` makes this safe.

### Wall time

Per-cell wall time assumption (cold-start dominates over inference on these
short tasks; sandbox boot adds ~3-5 s per cell):

| Model class | Per-cell wall | Cells | Concurrency | Wall time |
|---|---|---|---|---|
| local (2 models × 96 cells = 192 cells) | ~30 s | 192 | 1 (serial, GPU lease) | **~1.6 hr** |
| cloud (3 models × 96 cells = 288 cells) | ~15 s | 288 | 2-wide (conservative) | **~0.6 hr** |
| trajectory judge (40 calls) | ~5 s | 40 | 2-wide | **~0.0 hr** |
| **End-to-end** (serial groups, locals then clouds, then judge) | — | 480 | — | **~2.3 hr** |

Slack: gVisor cold-boot overhead, retries, and any partial-quota throttle
on the cloud side easily push this to 3-4 hr in practice. Budget **3-4 hr**
for the generation sweep, **30 min** for deterministic eval, **15 min** for
the 40-call judge slice. Total Day-1 wall time: **~4-5 hr**.
